Small plasmids recovery issue

an unexpected issue emerged in the assembly assessemnt: For the same sample and the same tool combination, 50x and 100x depth assemblies sommetimes miss small plasmids that the 20x depth dataset successfully recovers.

I perfomrd the following initial investigation:

  • Breadth of coverage Verification: When the high depth datasets missed the small plasmids but the low depth recovered them, the low depth sample still had excellent base coverage (>92%).

  • This “high depth misses, low depth recovers” issue is entirely exclusive to the E. coli sample, to be precise is contig plasmid_4 (1919bp). Pairwise alignment confirmed that plasmid_4 does not share high sequence similarity (~35%) with plasmid_4 (3173bp).

So I further just focus investigating E coli datasets only.

  • for hac, chopper-porechop_abi 50x in E coli plasmid_4, BPH2947__202310 plasmid_2 (3011bp, 39.1232% cov) chopper-porechop_abi 100x (partial). seqkit-dorado plasmid_2 (3011bp, 38.9904% cov) on BPH2947__202310.

  • Algorithm Determinism: To ensure this was not caused by Flye’s randomness, I manually reran Flye 10 times for two processed datasets (seqkit-barbell and seqkit-dorado) and unprocessed-untrimmed dataste across the three downsampling depths. Eech dataset always give the same result whteher it recovered the sample or no, it always give the same result for all 10x iterations.

  • So, I think this discrepancy is not due to randomness in the Flye algorithm decision, but is inherently tied to how the underlying Flye algorithm processes high copy number and short plasmid sequences.

Fly Struggle with small plasmids

My premise : Flye is inherently struggle with small plasmid recovery.

Finding from literatures and github issue

There are mutiple evidence or finding that oftenly Flye stuggle to recover small plasmid, especially in the high depth sample:

-   There is ongoing issue about small plasmid recovery on Flye https://github.com/mikolmogorov/Flye/issues/706

- High depth impact small plasmid recovery (https://www.microbiologyresearch.org/content/journal/mgen/10.1099/mgen.0.001024#F3). Copy number increase, and just too high, and flye treat them as "noise" maybe from non biological seqeunce such as adapter or barcode?

import os
import glob
import pandas as pd
import seaborn as sns
import matplotlib.pyplot as plt
from matplotlib.ticker import ScalarFormatter
from pathlib import Path

# --- 1. Define the Color Palette ---
named_colors = {
    "black": "#000000", "orange": "#e69f00", "skyblue": "#56b4e9", 
    "vermilion": "#d55e00", "bluish green": "#009e73", 
    "yellow": "#f0e442", "blue": "#0072b2", "reddish purple": "#cc79a7",
}
cud_palette = list(named_colors.values())

def cud(n=len(cud_palette), start=0):
    return (cud_palette[start:] + cud_palette[:start])[:n]

# --- 2. Data Parsing ---
search_pattern = "../results/assembly/*/*/*/ATCC_25922__202309.*.assembly_info.txt"
files = glob.glob(search_pattern)

records = []
for f in files:
    path = Path(f)
    model = path.parent.name
    depth = path.parent.parent.name
    combo = path.parent.parent.parent.name

    try:
        df = pd.read_csv(f, sep='\t')
        # Check for both length and cov. columns (Flye uses 'cov.')
        if not df.empty and 'length' in df.columns and 'cov.' in df.columns:
            # Filter for contigs strictly less than 6000 bp
            small_contigs = df[df['length'] < 6000]

            for _, row in small_contigs.iterrows():
                records.append({
                    "combo": combo,
                    "depth": depth,
                    "model": model,
                    "length": row['length'],
                    "coverage": row['cov.']
                })
    except pd.errors.EmptyDataError:
        continue

df_plot = pd.DataFrame(records)

# --- 3. Plotting Setup ---
sns.set_theme(style="whitegrid")
models = ["sup", "hac"]
hue_order = ["100x", "50x", "20x"]

fig, axes = plt.subplots(nrows=1, ncols=len(models), figsize=(14, 6), dpi=300, sharex=True, sharey=True)

# Plasmid multimer targets (up to 3x, since limit is 6000bp)
target_lengths = [1919, 1919*2, 1919*3]

for c, model in enumerate(models):
    ax = axes[c]
    df_sub = df_plot.query("model == @model")
    
    if not df_sub.empty:
        # Changed to scatterplot for continuous X and Y data
        sns.scatterplot(
            data=df_sub, x="coverage", y="length", hue="depth", 
            hue_order=hue_order,
            palette=cud(len(hue_order), start=2), 
            ax=ax, alpha=0.7, edgecolor="black", s=60 # 's' controls dot size
        )
        
    # Draw horizontal target lines for the multimers
    for i, t_len in enumerate(target_lengths):
        ax.axhline(y=t_len, color='red', linestyle='--', alpha=0.5, zorder=0)
        if c == 1: 
            # Placed text label at an arbitrary high coverage spot (e.g., x=10000) so it floats on the right
            ax.text(10000, t_len + 50, f"{i+1}x ({t_len}bp)", color='red', alpha=0.7, fontsize=8)

    # --- NEW: Apply Log Scale to X-Axis ---
    ax.set_xscale("log")
    # Force the log scale to show actual numbers (10, 100, 1000) instead of scientific notation (10^1, 10^2)
    ax.xaxis.set_major_formatter(ScalarFormatter())
    
    # Formatting
    ax.set_title(f"Coverage vs Length (<6kb) - {model.upper()} Model")
    if c == 0:
        ax.set_ylabel("Contig Length (bp)")
    else:
        ax.set_ylabel("")
        
    ax.set_xlabel("Contig Coverage (Log Scale)")
    ax.xaxis.grid(True, linestyle='--', color='lightgrey', zorder=0)
    
    # Restrict Y-axis limit strictly to 6000
    ax.set_ylim(0, 6000)
    
    # Give the X-axis some breathing room so points don't hug the edges
    ax.set_xlim(1, 20000) 

    # Manage Legends
    if c == len(models) - 1 and ax.get_legend() is not None:
        ax.legend(title="Depth", bbox_to_anchor=(1.05, 1), loc='upper left')
    elif ax.get_legend() is not None:
        ax.get_legend().remove()

fig.tight_layout()
output_file = "../results/figures/assess/assembly-combo-100length/metrics/plasmid_coverage_vs_length_6kb.png"
fig.savefig(output_file, bbox_inches='tight')
plt.close(fig)
print(f"Plot saved successfully to {output_file}")

Investigate how exactly Flye build the assembly

I read through this paper on how Flye solve the assembly problem from long read data. In the part of disjointig generation step, when looking to overalp each reads and concatenate togther, they said:

Let Alignments = Alignments(Genome, minOverlap) be the set of all sufficiently long (of length at least ‘minOverlap’) self-alignments of a genome ‘Genome’. Flye sets the ‘minOverlap’ parameter as the N90 of the read-set (the N90 of reads is the largest possible number N such that all reads of length N or longer have a total length of at least 90% of the total sequence; ‘minOverlap’ varies from 3,000 to 5,000 nucleotides for the SMS datasets analyzed in this paper).

Most of the the assembly that missed the small contigs have fastq N90 higher than plasmid_4 length

I also manually checked: in the unprocessed datasets, plasmid_4 is consistently recovered at high depths, but it appears as a 3838 bp dimer or a 5757 bp trimer in the orther dataste.

I further tried mapping these dimer and trimer against refernence plasmid_4. I found that that dimer or trimer happens due to a 19-bp sequence (GTTAAGCAGTTCCCCAACT) that bridges the plasmids together: [plasmid_4] + [GTTAAGCAGTTCCCCAACT] + [plasmid_4]

These bridging sequence is not part of the plasmid_4 contig, and I also confirm that that sequence is also not barcode/adapter sequence

My premise is: The sequence come from raw reads, and whther it is reatiend or not just before assembly process it is caused the plasmid_4 dimer or trimmer collapse or not in the Flye repeat graph reconsturction.

If i look at the IGV screenshot below where i maped raw reads (unprocessed-trimmed in this case) agianst dimer plasmid_4 from the same dataste, the mapped bridging sequence is part of several reads (i counted them about 6% of total read mapped)
