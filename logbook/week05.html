Feature dorado trim porechop_abi barbell kit fastplong
Adapter input It requires one kit name (--sequencing-kit) It can detect adapter from the read set, de novo (--ab_initio). One adapter will be selected. It requires one kit name (--kit) Detect automatically, but the author recommends to specify the sequence (start_adapter, end_adapter and adapter_fasta)
Adapter trimming extension Not specified --extra_end_trim, (default on, = 2bp) Not specified --trimming_extension(default on, =10bp)
Middle adapter detection Not specified Yes, it can split into two reads (default, to disable searching = --no_split) or discard them ( --discard_middle). It seems –-use-extended option can do this ( detect fusions, breaks) No, it searches adapter at start and and of the reads
How does the tool find adapter Corresponding adapter sequence of the specified kit Sample the reads (--check_reads, default 10Kb)*, then will use kmer + graph based method to find consensus (--abi), or match them to static db (original impl) Corresponding adapter sequence of the specified kit Not clear. A static database seems to be expected (similar to original porechop), but the nanopore adapter is not included here in their knownadapters.h file. Maybe we need to specify adapter sequence afterall.
Tool related parameters -

--end_size (number of bases at each end of the read which will be searched for adapter sequences, default 150bp)

--end_threshold threshold identity (default 75%) to be removed and --adapter_threshold (default 90%) to be identitfied as present in the dataset when it is searching.

--maximize adds “risky” pattern during demultiplexing, meaning more reads are retained or assigned.

When scanning for barcode flanks, we can set --flank-max-errors. Other paramters related threshold --min-score,--min-score-diff, --alpha
Multithreading support (for benchmarking maybe?) --threads (deafult using all available threads) --threads (default is 16) --threads (default is 10) --thread (default is 3)

*: It doesn’t use proportion, meaning that different read depth will have different sampling representation to “guess” (ab initio) what the adapter sequence are. Is it better to perform downsampling before adapter trimming/QC steps instead of after to minimise bias across the samples?

Example commands

chopper (--trim-approach split-by-low-quality)

chopper-linux --trim-approach split-by-low-quality --cutoff 15 -l 50 -i biox7021/data/fastqs/KPC2__202310.fastq > split_reads.fastq
Kept 1862406 reads out of 49458 reads

fastplong

This command will only perform adapter trimming without quality and length filtering

fastplong -i {input.reads} -o {output.reads} --disable_quality_filtering --disable_length_filtering
